SQL Injection

L'injection SQL est classée #3 dans le Top 10 OWASP. C'est l'une des vulnérabilités les plus anciennes, les plus répandues et les plus dangereuses du web. Comprendre comment elle fonctionne est la première étape pour s'en protéger.

L'impact en chiffres

#3

65%

Comment ça marche ?

Une injection SQL se produit quand une application insère des données utilisateur directement dans une requête SQL sans les filtrer. L'attaquant peut alors modifier la requête pour accéder à des données non autorisées, contourner l'authentification, ou même supprimer des tables.

VULNÉRABLE
// L'entrée utilisateur est concaténée directement const sql = `SELECT * FROM users WHERE username = '${username}' AND password = '${password}'`;

Si l'utilisateur entre ' OR 1=1 -- comme nom d'utilisateur, la requête devient :

SELECT * FROM users WHERE username = '' OR 1=1 --' AND password = '...'

La condition OR 1=1 est toujours vraie, et -- commente le reste. Résultat : l'attaquant est connecté comme le premier utilisateur de la table (souvent l'admin).

Types d'injections SQL

1. Tautologie (OR 1=1)

Contourne l'authentification en rendant la condition toujours vraie.

' OR 1=1 --

2. UNION SELECT

Combine les résultats de la requête originale avec une requête de l'attaquant pour extraire des données d'autres tables.

' UNION SELECT 1,username,password,4,5,6 FROM users --

3. Extraction du schéma

Récupère la structure de la base de données (noms de tables, colonnes) pour planifier des attaques ciblées.

' UNION SELECT 1,name,sql,3,4,5 FROM sqlite_master --

4. Blind SQL Injection

Quand l'application ne montre pas les résultats SQL, l'attaquant pose des questions vrai/faux pour extraire les données caractère par caractère.

' AND SUBSTR(password,1,1)='S' --

Vulnérable vs Sécurisé

Concaténation (vulnérable)

const sql = `SELECT * FROM products WHERE name LIKE '%${query}%'`; db.prepare(sql).all();

Requête paramétrée (sûr)

const sql = `SELECT * FROM products WHERE name LIKE ?`; db.prepare(sql) .all(`%${query}%`);

Avec une requête paramétrée (?), la base de données traite la valeur comme une donnée, jamais comme du code SQL. L'injection est impossible.

Comment se protéger

  • ✓
    Requêtes paramétrées (Prepared Statements) La défense #1. Utilise des placeholders (? ou :name) au lieu de concaténer les variables. Disponible dans tous les langages et frameworks.
  • ✓
    ORM (Object-Relational Mapping) Les ORM comme Sequelize, Prisma ou SQLAlchemy génèrent des requêtes paramétrées automatiquement. Moins de risque d'erreur humaine.
  • ✓
    Validation des entrées Vérifie le type et le format attendu. Un ID doit être un nombre, un email doit correspondre au format email, etc.
  • ✓
    Principe du moindre privilège L'utilisateur de la base de données ne doit avoir que les permissions nécessaires. Pas de DROP TABLE si l'app ne fait que lire.
  • ✓
    Ne jamais exposer les erreurs SQL Les messages d'erreur détaillés aident l'attaquant. En production, affichez un message générique et loguez l'erreur côté serveur.
  • ✓
    WAF (Web Application Firewall) Couche de défense supplémentaire. Détecte et bloque les patterns d'injection connus. Ne remplace pas les bonnes pratiques de code.

Attaques célèbres

  • Sony Pictures (2011) — 77 millions de comptes compromis via injection SQL sur le PlayStation Network.
  • Heartland Payment Systems (2008) — 130 millions de numéros de cartes volés. Coût : 140 millions de dollars.
  • TalkTalk (2015) — 157 000 clients affectés. Un adolescent de 17 ans a exploité une simple injection SQL.
