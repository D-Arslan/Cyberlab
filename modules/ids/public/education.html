IDS / IPS : Theorie et pratique

Comprendre les systemes de detection et de prevention d'intrusion, et apprendre a ecrire des regles Snort efficaces.

1. Qu'est-ce qu'un IDS ?

Un IDS (Intrusion Detection System) est un systeme qui surveille le trafic reseau en temps reel et detecte les activites suspectes ou malveillantes.

Analogie : Un IDS est comme une camera de surveillance. Il observe tout ce qui passe et declenche une alarme quand il voit quelque chose de suspect. Mais il ne bloque rien.

IDS vs IPS

IDS (Detection) IPS (Prevention)
Role Detecte et alerte Detecte et bloque
Position En copie (mirroring) En ligne (inline)
Impact Aucun sur le trafic Peut bloquer du trafic legitime
Latence Aucune Legere latence ajoutee
Risque Attaque non bloquee Faux positifs = blocage de trafic normal

Types de detection

  • Par signature : Compare le trafic a une base de regles connues (comme un antivirus). Rapide mais ne detecte pas les attaques inconnues.
  • Par anomalie : Etablit un profil "normal" du trafic et detecte les deviations. Detecte les attaques inconnues mais genere plus de faux positifs.
  • Hybride : Combine les deux approches. C'est ce que font les solutions modernes.

2. Snort en detail

Snort est l'IDS/IPS open-source le plus utilise au monde. Cree en 1998 par Martin Roesch, il est maintenant maintenu par Cisco.

Les 3 modes de Snort

  • Sniffer : Affiche les paquets en temps reel (comme tcpdump)
  • Packet Logger : Enregistre les paquets dans des fichiers log
  • IDS/IPS : Analyse le trafic avec des regles et genere des alertes

Architecture de Snort

Packet Decoder → Capture et decode les paquets bruts

Preprocessors → Recompose les flux, normalise le trafic

Detection Engine → Compare les paquets aux regles

Output Plugins → Genere alertes, logs, envoie vers SIEM

Commandes essentielles

# Tester la configuration sudo snort -T -c /etc/snort/snort.conf # Lancer en mode IDS (alertes dans le terminal) sudo snort -A console -q -c /etc/snort/snort.conf -i eth0 # Lancer avec logs dans un fichier sudo snort -A full -l /var/log/snort/ -c /etc/snort/snort.conf -i eth0 # Lire un fichier pcap sudo snort -r capture.pcap -c /etc/snort/snort.conf

3. Anatomie d'une regle Snort

Une regle Snort se compose de deux parties : le header (qui definit quoi matcher) et les options (qui affinent la detection).

alert tcp 10.0.0.0/8 any -> 192.168.1.0/24 80 (msg:"SQLi detectee"; content:"OR 1=1"; sid:1000001;)
Action
Protocole

Options

Actions possibles

  • alert : Genere une alerte et log le paquet
  • log : Log le paquet sans alerte
  • pass : Ignore le paquet
  • drop : Bloque le paquet (mode IPS uniquement)

Options principales

Option Description Exemple
msg Message d'alerte msg:"Ping detecte";
content Cherche une chaine dans le payload content:"OR 1=1";
flags Flags TCP (S=SYN, A=ACK, F=FIN...) flags:S;
sid ID unique de la regle sid:1000001;
rev Numero de revision rev:1;
priority Priorite de l'alerte (1=haute) priority:1;

4. Types d'attaques detectables

Scan de ports (Reconnaissance)

L'attaquant envoie des paquets SYN vers differents ports pour decouvrir les services actifs. Outil typique : nmap.

# Regle Snort pour detecter un scan SYN alert tcp any any -> any any (msg:"Scan SYN detecte"; flags:S; sid:1000010;)

Ping Flood (DoS)

Envoi massif de paquets ICMP Echo Request pour saturer la bande passante de la victime.

# Regle pour detecter un ping flood alert icmp any any -> any any (msg:"Ping flood possible"; sid:1000011;)

SQL Injection

Insertion de code SQL malveillant dans des requetes HTTP pour manipuler la base de donnees.

# Regle pour detecter une injection SQL basique alert tcp any any -> any 80 (msg:"SQL Injection tentative"; content:"OR 1=1"; sid:1000012;)

Brute Force SSH

Tentatives repetees de connexion SSH avec differents mots de passe.

# Regle pour detecter des tentatives SSH alert tcp any any -> any 22 (msg:"Tentative SSH suspecte"; content:"SSH"; sid:1000013;)

Exfiltration DNS

Utilisation du protocole DNS pour envoyer des donnees volees vers un serveur controle par l'attaquant (C2).

# Regle pour detecter l'exfiltration DNS alert udp any any -> any 53 (msg:"DNS suspect vers domaine malveillant"; content:"evil.com"; sid:1000014;)

5. Bonnes pratiques

Regle d'or : Une bonne regle IDS detecte toutes les attaques (pas de faux negatifs) sans declencher d'alertes sur le trafic normal (pas de faux positifs).

  • Etre specifique : "alert tcp any any -> any 80" est trop large. Ajouter des criteres (content, flags) pour reduire les faux positifs.
  • Tester avant de deployer : Utiliser des fichiers pcap pour valider les regles avant la mise en production.
  • Maintenir a jour : Les regles communautaires Snort sont mises a jour regulierement avec les nouvelles signatures d'attaques.
  • Combiner avec d'autres outils : Snort + SIEM (Splunk, ELK) pour la correlation et l'analyse.
  • Documenter : Chaque regle custom doit etre documentee (pourquoi, quand, par qui).
  • Surveiller les performances : Trop de regles = latence. Prioriser les regles critiques.

Faux positifs vs Faux negatifs

Faux positif : L'IDS alerte sur du trafic normal. Resultat : fatigue d'alerte, les vrais incidents sont ignores.

Faux negatif : L'IDS laisse passer une vraie attaque. Resultat : l'attaquant agit sans etre detecte.

Equilibre : Trop de sensibilite = faux positifs. Pas assez = faux negatifs. Il faut trouver le juste milieu.
